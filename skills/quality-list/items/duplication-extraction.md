# Duplication extraction

When a diff adds a **candidate** — a construct or block near-identical to at least one existing instance (in the diff or the codebase), or matching a scaffolding shape (inline fixture setup, repeated literal scaffolding, parallel branches differing only in a value) — the trigger has two branches. If the existing-helper search below finds a helper that already expresses the construct and is callable from the site, the first added copy is the finding: propose calling that helper. If no such helper exists, the Nth (N ≥ 3 across the file or module) near-identical construct is the finding: propose extracting a new helper (a function, a fixture factory, a table-driven loop, a parametrized test). The default disposition is **actionable**; it is not downgraded by appeal to the surrounding code, because neighboring duplication is unextracted debt, not a convention to match. In the extraction branch, the pre-existing copies are what make this the Nth occurrence — they are the trigger, not an exemption.

The rule of three (tolerate two occurrences; extract on the third) is the threshold heuristic the extraction branch rests on. Below three, abstracting risks the wrong abstraction — coupling code that only looks alike. It does not apply to the existing-helper branch: there the abstraction already exists, so there is no premature-abstraction risk to guard against, and calling the helper is right from the first copy.

**Existing-helper search — concept first, pattern second.** For each candidate, before counting instances, name what the construct is *for* — "build an identity operator", "parse a duration", "normalize a path" — and search for that concept by name across the current file, the enclosing module's private helpers, symbols reachable through imports, and the library / shared modules. Only when the concept search finds no usable helper does a sub-three count of near-identical text instances mean what the N/A criterion below takes it to mean — no existing helper covers this, and the duplication is below the extraction threshold: a helper that does the same job with different literals, a different internal spelling, or a different decomposition is invisible to a text-pattern search, so a count produced without the concept search is a property of the query, not of the codebase. Candidates sharing one named concept are covered by a single recorded search. Record the queries — the concept terms and the pattern grep — alongside the count in the evidence; a bare count is unfalsifiable.

**Concern conditions:**

- The concept-first search finds an existing helper, callable from the construct's site, that could express a candidate the diff adds inline, and the site is not deliberate independence (the N/A carve-out below) — a finding from the first added copy. (A found helper that cannot permissibly be made callable from the site — a boundary rule forbids the import (`architectural-boundary`'s territory), or the needed visibility widening is disallowed — is no usable helper; the candidate falls to the extraction branch. A helper reachable by adding an allowed import or dependency edge is callable: proposing that edge is part of the finding.)
- A diff adds an Nth (N ≥ 3) near-identical construct or block that a trivially-extractable helper could express, and the concept-first search found no existing usable one.

**N/A:**

- The diff adds no candidate: no added construct or block is near-identical to an existing instance or matches a scaffolding shape. No search obligation arises.
- The inline copy is deliberate independence: calling the helper would make the site's correctness depend on the very code the site exists to check independently — a test's expected value hand-built so the assertion is not a helper-vs-helper tautology, a differential fixture. The exemption covers only calling that helper; at N ≥ 3 the copies remain extraction-branch candidates for a new independence-preserving helper (a test-side fixture factory), and the exempted helper does not count as a found usable one for the extraction branch's purposes.
- Fewer than three near-identical instances exist, the concept-first existing-helper search was run and found no usable helper (none, or only helpers ruled out — not permissibly callable from the site, or not actually expressing the construct), and the queries are recorded. A sub-three count produced without that search does not discharge the item — the count is a property of the search that produced it, not of the codebase.
- The constructs are only superficially similar, or no single new helper cleanly expresses them without coupling unrelated code or restructuring control flow beyond the mechanical rewrite into one of the helper forms above (rewriting the duplicated branches themselves into a table-driven loop is the extraction, not the disqualifier).
- The repetition is defensive-transformation replication repairing invalid producer output — that is `public-api-surface` Concern A's territory, where the fix is tightening the producer API rather than extracting a helper.
