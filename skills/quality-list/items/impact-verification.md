# Impact / caller verification

If the change has a planned impact list (from research or design notes), verify it against the actual diff:

- Every caller listed as affected has been updated (gap = missed impact)
- No caller has been modified that wasn't in the impact list (gap = scope creep)

When no formal impact list exists, manually trace the public symbol's callers and confirm each remains consistent with the change.

Beyond signature compatibility, check for **value-class shifts**: a change — with or without a signature change — can make a value class newly reachable or unreachable in a symbol's returns. Value classes are: zero / non-zero, negative / non-negative, empty / non-empty, `None` / `Some`, NaN / finite, subnormal / normal, the enum-variant set, and bounds stated on the symbol itself (its own docstring or type) — the boundaries consumers typically branch on. Read the shift from diff-visible evidence (an added or removed guard, clamp, or early return; or an operation whose own form changes the reachable result class — a newly introduced division that can yield infinity, a subtraction that can now produce a negative), not whole-program value-range analysis and not per-operation class speculation over every touched expression; a value move within the old classes (an off-by-one count fix), or an arithmetic-only refactor that reorders or fuses operations without touching a guard/clamp/early-return or introducing a class-changing operation, is not a class shift, unless the resulting perturbation itself crosses a class boundary (an exactly-zero result becoming tiny non-zero). When a class boundary shifts, the caller trace must additionally check consumers' class-dependent behavior at the class(es) added or removed: reciprocals and guards at a newly-reachable zero or subnormal, match exhaustiveness and emptiness checks at a new variant or newly-empty result. Trace direct consumers; when a direct consumer only stores or returns the value unchanged, follow it one hop to that surface's consumers, and record deeper propagation as the trace boundary instead of chasing it. Documentation claims invalidated by the same shift — stated bounds and closed enumerations alike — are `docstring-drift`'s concern; this item owns consumer code.

Also check for **value-set-preserving contract shifts**: a same-signature change can leave the eventually-produced value set intact while shifting the contract consumers rely on — the same elements in a different order, or evaluation timing shifted between eager and lazy (including the point where an error is raised moving between call and first consumption). Read the shift from diff-visible evidence — an ordering operation added, removed, or relocated (a sort, reverse, or comparator; an ordered container or query clause exchanged with an unordered one); or a materialized result exchanged with a deferred one (a generator, iterator, stream, or closure exchanged with a built collection) — not speculation that a refactor "might" change order or timing. When such a shift is visible, the caller trace must additionally check consumers' sensitivity to the shifted dimension: order-sensitive consumption (binary search, positional indexing, first/last takes, zip against a parallel sequence) at an ordering shift; re-iteration, exhaustion, and error-handling placement at a timing shift. The trace bound is the same return-value trace as for value-class shifts. As with value-class shifts, documentation claims invalidated by the shift — a stated ordering, a documented evaluation-timing guarantee — are `docstring-drift`'s concern.

Also check for **side-effect and result-identity contract shifts**: a same-signature change can leave the returned value intact while adding, removing, reordering, or relayering a side effect, or changing result identity — adding or removing memoization, or swapping a returned reference between a shared object and a copy or freshly constructed instance. For this trigger, use the **observer-bound trace** defined here — not the return-value trace the two shifts above use — since an effect's observers are generally not consumers of the return value, so a return-value trace neither finds them nor bounds the search. First apply a **materiality gate**, decided before any trace and by the effect's *type* rather than by traced observability: an argument or shared-state mutation, a persistence / network / global-state effect, or a result-identity change (a memoization toggle or a shared-vs-copy return swap) is material; diagnostic-only logging to a log sink is immaterial — keying materiality to the effect category, not to whether some consumer can observe it, avoids the circularity of exempting an effect by the very trace the exemption would gate. Then run the observer-bound trace, bounded up front: its core is each direct caller's post-call reliance at the call site — whether the caller reads a mutated argument after the call, depends on the write or memo having happened, or assumes the effect's absence; its boundary is distant observers of the shared state or persisted data, recorded by the **state surface** they touch (the mutated global, the table, the file path, the network destination — an endpoint, topic, or queue) rather than by enumerating observers — record the surface as the trace bound instead of chasing every reader. Sensitivity is asymmetric by verb: an added effect breaks callers relying on its absence, a removed effect breaks callers relying on it having happened, and a reordered effect breaks callers relying on effect order; a result made more shared — memoization added, or a copy replaced by the shared object — breaks callers that mutate the returned value or need a fresh isolated instance (aliasing), and a newly memoized result additionally breaks callers relying on recomputation against changed state (staleness); a result made fresher — memoization removed, or the shared object replaced by a copy or fresh construction — breaks callers relying on result-identity stability or on shared-mutation propagation. Read the shift from diff-visible evidence — a mutation, write, network call, or log statement added or removed; a memo or cache wrapper added or removed, or a returned reference swapped between a shared object and a copy or freshly constructed instance; an effect statement moved across the return or across another effect — not speculation that a refactor "might" change effects. As with the shifts above, documentation claims invalidated by the shift are `docstring-drift`'s concern; this item owns consumer code.

Also check for **input-acceptance domain shifts**: a change can replace a parsing, decoding, or input-conversion primitive with another (`std::strtod` → `std::stod` — the new `std::stod` rejects under libc++ the subnormals the old `std::strtod` accepts; a lenient numeric parse for a stricter one; one decoder for another) without changing the consuming symbol's signature, shifting the set of inputs it accepts. This is the input-side dual of the value-class shift above: instead of a value class becoming newly reachable or unreachable in a symbol's *returns*, a value class becomes newly readable or unreadable in a symbol's *accepted inputs*, so the trace runs **upstream to the producers whose emitted set feeds this consumer**, not downstream to callers. Read the shift from diff-visible evidence — one parse / decode / convert primitive literally swapped for another — not speculation that a refactor "might" narrow acceptance. When such a swap is visible, state the old primitive's accepted domain and the new one's and characterize their symmetric difference by its value classes, representation forms, and boundary cases — not necessarily every member, since the difference may be an infinite family (arbitrary surrounding whitespace, additional numeric spellings): a class the old primitive accepted and the new one rejects is a regression the motivating bug hides — the producer still emits it and it is now unreadable — while a class the new one accepts and the old rejected is a widening rarely stated; where a primitive's accepted set is implementation- or locale-defined (as `operator>>` and `strtod`'s underflow handling are), name that dependence as part of the difference rather than treating it as an exemption, since an implementation-defined edge widens the risk set. Then trace each direct producer feeding this consumer — the paired writer / serializer for a parser, the upstream converter for a converter — and check both the value classes and the representation forms it can actually emit (its serialization format — surrounding whitespace, leading zeros, sign, quoting, precision — since a swap often rejects a spelling while the semantic value class is unchanged) against the symmetric difference — a class or form the producer cannot emit needs no case, so the check is scoped to what the producer can actually produce; when a producer only forwards a value unchanged, follow one hop upstream and record deeper propagation as the trace boundary. When a producer is external to the codebase and cannot be traced, treat every class or representation form the new primitive newly rejects as live risk, since an untraceable producer cannot be assumed not to emit it. That upstream trace resolves the narrowing case; a widening resolves downstream instead — inputs that used to be rejected now succeed, so trace consumers that relied on the old rejection (error handling, validation, fallback), whether or not the newly-accepted input yields a new output value class; when it does yield a new class, that is additionally the value-class shift in returns covered above, traced downstream to this consumer's callers. As with the shifts above, documentation claims invalidated by the swap are `docstring-drift`'s concern; this item owns producer and consumer code.

**Concern conditions:**

- A listed caller was not updated
- A caller was updated but is not in the impact list (or the deviation is not justified)
- Public symbol changed but no caller trace was performed
- The diff made a value class newly reachable or unreachable in a symbol's returns, and the caller trace did not check class-dependent consumer behavior at the shifted class(es) within the return-value trace bound above
- The diff shows a value-set-preserving contract shift, and the caller trace did not check consumer sensitivity to the shifted dimension within the return-value trace bound above
- The diff adds, removes, reorders, or relayers a material side effect, or changes result identity (memoization added or removed, or a shared-vs-copy return swap), and the caller trace did not check observer sensitivity to the shift within the observer-bound trace bound above
- The diff swaps a parsing / decoding / conversion primitive for another without changing the consuming symbol's signature, and the trace did not state the accepted-domain symmetric difference or did not check each upstream producer's actually-emitted value classes and representation forms against what the new primitive newly rejects — or, for inputs the new primitive newly accepts, did not trace consumers relying on the old rejection (and, when a new output class results, the value-class shift in returns per the condition above)

**N/A:** the change touches no symbol with cross-module callers (internal helper with single use site, isolated test, etc.), and no value-class shift, value-set-preserving contract shift, side-effect / result-identity contract shift, or input-acceptance domain shift is diff-visible. A class shift, contract shift, effect shift, or acceptance shift whose bounded trace finds no affected cross-module consumer, observer, or producer is a pass with evidence, not N/A.
