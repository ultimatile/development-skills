# Escape-hatch / workaround necessity

A workaround — any construct that routes around a defect or limitation instead of fixing it directly — is not self-justifying. Static-guarantee escape hatches are the representative class (Rust `unsafe`, C++ `reinterpret_cast` / `const_cast` / C-style cast, TypeScript `any` / `as` / non-null `!` / `@ts-ignore`), but the same obligation binds a sleep standing in for a missing readiness signal, a retry loop papering over a race, a special-case branch absorbing an upstream defect, or a defensive re-check compensating for an invariant the producer should guarantee. In every case the workaround's **necessity** must be derived before it is accepted: show that the direct fix is genuinely out of reach. A justification comment (Rust `// SAFETY:`, a cast rationale, a "this is fine because…" note) argues the workaround is *sound* — why it does not break; it does not argue it is *necessary* — why no direct fix can remove the problem. Soundness and necessity are separate obligations, and the comment discharges only the first.

The recurring failure is a framing error. A problem solvable by fixing the underlying cause — placing behavior where the type system already knows the concrete type (trait / interface dispatch, generics, an enum), adding the missing readiness signal, repairing the producer's invariant — gets reframed as a *local mitigation* problem and solved by the workaround. The workaround compiles and a sound-looking justification can be written, so it reads as "solved" locally — but the direct fix would have removed the problem setting entirely, justification and all. Hence the heuristic "if you need a paragraph-long comment to justify why the workaround is OK, the code is wrong — fix the code": a justification that grows to a paragraph is a signal that effort is flowing into defending acceptability rather than into deriving necessity. A signal only, not a verdict — when the explanation establishes a genuine external constraint, the long comment is exactly right. Either way the comment is evidence, not the trigger: a workaround carrying no justification at all is *further* from establishing necessity, not exempt from it.

Trigger when a diff introduces a workaround whose necessity is not established by the diff and its available context. Ask: can a direct fix remove the problem setting instead? If yes, the workaround is unnecessary, not merely sound — and "sound but unnecessary" is the defect.

**Concern conditions:**

- Diff introduces a workaround whose necessity is not established (per the trigger above) — the accompanying justification, if any, argues acceptability only
- The representative instance of the above: diff adds an escape hatch to convert between types a safe construct (trait dispatch, generics, enum) could keep concrete at the use site

**N/A:** the workaround sits at an irreducible external constraint that no direct fix can remove, and the constraint is identifiable — from the construct itself (FFI / `extern` call, raw allocation, hardware MMIO, a layout-compat pointer cast guarded by static size / align assertions) or from the justification (a platform or protocol requirement, an upstream defect that is linked *and* explained — a bare link is insufficient; the text must state why the external defect forecloses the direct fix here). There, necessity is established and only soundness remains to argue. The discriminator against the deferral case below is *who can remove the constraint*: N/A applies only when no fix within this codebase's control can remove it. (And a retry with backoff around an inherently fallible external operation — network I/O against a remote service — routes around nothing; it is not a workaround in the first place.)

A recorded scope deferral — an issue link or plan decision carrying a reason and a resolution point for a direct fix that *is* within this codebase's reach and already understood — does not make the workaround *necessary*; it makes the debt *acknowledged*. Treat such a case per `finding-triage`'s `defer` disposition, never as N/A; a deferral whose fix still needs investigation, redesign, or a non-local (cross-component) change is `opens-a-question`, not `defer`. In an audit this surfaces as a concern closed by recording the deferral (a follow-up issue filed with the user's approval) rather than by fixing now — `defer` refines how the concern is closed out; it does not add a new verdict to any runner's vocabulary.
